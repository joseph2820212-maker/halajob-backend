How flags work

Every feature flag lives in the backend's .env and is surfaced to clients via GET /public/v1/client-settings. Clients read the flags at boot; changing a flag on the backend requires clients to refetch settings (all three clients do this at app start).

Naming convention: FEATURE_<NAME>_ENABLED=true|false. A few flags carry a value (FEATURE_PAYMENTS_MODE=manual|online) — treat those as enum flags.

AI provider deep-dive

AI is disabled by default. Turning on FEATURE_AI_TOOLS_ENABLED without configuring a provider will surface features that fall back to cv_parser_not_configured / ai_provider_not_configured errors to the client.

<code>HALA_AI_ENABLED</code>
Master AI switch on the provider layer. Must be true for anything downstream.
<code>HALA_AI_PROVIDER</code>
Provider slug (owner supplies).
<code>HALA_AI_MODEL</code>
Model name.
<code>HALA_AI_API_KEY</code>
Provider API key.
<code>HALA_AI_BASE_URL</code>
Provider base URL. Defaults to OpenAI.
<code>HALA_AI_TIMEOUT_MS</code>
Request timeout (default 30 s).
<code>HALA_AI_DAILY_LIMIT</code> / <code>_MONTHLY_LIMIT</code>
Per-account rate cap on AI calls.
<code>HALA_AI_ALLOW_MOCK_PROVIDER</code>
Local verification only. Prod ignores this unless explicitly true. Mock output must never be labelled real AI.
Do not label mock AI as real AI. This is a handoff rule. The mock provider is for local dev only; production output must be from a live provider.

Payments deep-dive

FEATURE_PAYMENTS_MODE is the single most product-visible flag. Two supported values:

  • manual — admins assign / adjust subscriptions in the admin console; there is no online checkout. This is the launch default.
  • online — payment provider integration. Currently no provider is enabled; enabling this without a provider will cause checkout flows to fail.
Do not label manual/admin subscriptions as online payment. This is a handoff rule.
