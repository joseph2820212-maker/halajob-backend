The product in one paragraph

Hala Job is a Syria-first hiring platform built for the reality of the market: bilingual (Arabic + English) by default, mobile-heavy, with a manual-payment mode that lets subscriptions launch before an online provider is in place. Job seekers, students, employers, and universities each get a dedicated app — all backed by a single, hardened API.

What ships today

Customer website

Public jobs, campus, seeker, and employer portals in one deployment. Green-lit at halajob.com; unknown paths safely fall back to home.

Admin console

Dedicated operator app on its own domain. Client auth gate — no admin content renders in the browser until an admin session exists.

Mobile app (Flutter)

Portrait-first native app for seekers and campus students. Offline-tolerant, obfuscated release builds, cleartext blocked at the OS.

Backend API

Express + MongoDB. Rate limits, CORS allow-list, mongo-sanitize, hpp, per-file upload guards. Zero dependency vulnerabilities in production audit.

Why the split matters

Ship any part of the product on its own schedule.

The four apps deploy independently. A design tweak to the customer site never waits on a backend release. A backend hotfix never triggers a Play Store submission. That means faster iteration, cheaper CI, and a smaller blast radius when something goes wrong.

Foundation

Technology &amp; security

The things a technical due-diligence check will actually inspect. Everything referenced here is in the shipped code and enforced in CI.

helmet + HSTSStrict CSPOS-level HTTPS0 known CVEsImmutable audit

Bilingual by default

Every schema stores content as { en, ar }. Every client picks language at boot. Every backend response respects x-lang.

Syria-first currency &amp; comms

SYP + USD + EUR supported at the launch contract level. Manual WhatsApp share enabled; SMS and official WhatsApp gated behind provider setup.

Hardened backend

helmet + HSTS preload, CORS allow-list with 403 on unknown prod origins, per-prefix rate limits (auth 20/15min, upload 60/15min), mongo-sanitize, HPP guard, upload size & type limits, per-file CV-download tokens with expiry.

Hardened clients

Website + admin ship a strict CSP with no unsafe-eval; inline scripts pinned by sha256 (drift caught in CI). Admin adds an extra client-side auth gate on top of backend 401s.

Hardened mobile

Cleartext blocked at the OS via network_security_config.xml; obfuscated release APKs with private symbols archive; secure token storage (EncryptedSharedPreferences / Keychain).

Immutable audit trail

Every admin state change is audited. Per-category TTL retention. No PII stored in audit rows.

Zero known-CVE production dependencies

npm audit --omit=dev --audit-level=high is a CI gate on all three JavaScript repos. Green today; a new advisory blocks the next PR.

Feature-flag envelope

16 backend flags surfaced to clients through a single GET /public/v1/client-settings — every launch decision is one config change, not a deploy.

Deploy independence

Four independent CI pipelines. A design tweak on the website never triggers an APK build. Admin CI is under a minute. Mobile APK jobs only run on manual dispatch.

What is launch-ready today

  • All four applications ship, type-check, test, build, and audit green in CI.
  • Backend serves 4,004 endpoints across seeker, company, campus, university, admin, trust, notifications, AI, analytics, and public modules.
  • Bilingual content across all four applications with legal, help, FAQ, contact, accessibility, privacy, support, and report surfaces.
  • Manual/admin subscriptions ship on day one. Online payments require an owner-selected provider — the switch is one config flag.
  • AI features ship disabled by default; every AI code path degrades gracefully to a "not configured" response.

What still needs the owner (not the engineers)

  • Production secrets rotation, live-database admin audit, and a live production smoke test.
  • Real-account setup for SMTP, Firebase/push, storage, hosting, and backup/restore.
  • Decision on payments (accept manual for launch, or select an online provider).
  • Decision on Android signing/package/update strategy.
Bottom line

The platform is buildable, launchable, and hardened. What is left is business decisions and operator credentials — not code.

The engineering delivery is done. Every launch blocker in the security review is an owner-side choice or a credential the platform genuinely cannot self-provision.
