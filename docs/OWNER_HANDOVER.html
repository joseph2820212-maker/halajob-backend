Launch gates + handoff rules

Owner-only launch gates

  1. Rotate production secrets. Replace every value in the previous secrets list with fresh values. Archive the previous set for rollback.
  2. Audit privileged users. Run npm run security:audit-users against the live database. Confirm every account with the admin role or elevated permissions matches an approved operator.
  3. Live production smoke. Run the deployment runbook end-to-end against the real prod environment (not staging). Health, login, one write, one read.
  4. Payments decision. Confirm launching with manual/admin subscriptions, or provide the online payment provider setup.
  5. Android signing decision. Confirm the signing strategy (upload key vs Play App Signing) before the first public APK.
  6. Backup drill. Run one full mongodump → restore into a scratch database. Time it; document the RPO/RTO you actually observed.
  7. Legal content approval. Set LEGAL_CONTENT_ENFORCEMENT_MODE=production; verify every required legal page has legalReviewStatus: approved (the admin content tab surfaces this).

Do-not-do handoff rules

  • Do not commit .env, keystores, Firebase JSON, uploads, logs, or backups.
  • Do not label mock AI output as real AI. The mock provider is dev-only.
  • Do not label manual/admin subscriptions as online payment.
  • Do not distribute an old APK as if it were rebuilt from the current source.
  • Do not claim production smoke, provider integrations, admin audit, secret rotation, or real-device mobile approval without proof.

Post-launch cadence (suggested)

  • Weekly: review audit_logs for anomalies; check npm audit and flutter pub outdated in CI runs.
  • Monthly: restore drill (backup restore into scratch); token rotation review; open-CVE sweep.
  • Quarterly: re-audit admin users; review CSP connect-src against actual traffic; refresh deploy runbook.
